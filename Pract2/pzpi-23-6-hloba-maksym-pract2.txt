2 ЗАВДАННЯ
     Метою практичного заняття є поглиблене вивчення архітектурних підходів у сучасних складних програмних системах на прикладі сімейства моделей Gemini. В межах роботи необхідно проаналізувати структурну організацію системи, принципи масштабування, взаємодію компонентів та використані архітектурні стилі. Завдання передбачає виконання комплексного аналізу архітектури Gemini як розподіленої системи, ідентифікацію ключових стилів побудови ПЗ та детальний опис ролей основних підсистем. Особлива увага приділяється дослідженню механізмів масштабування та відмовостійкості у хмарному середовищі Google Cloud. Результатом роботи має стати підготовка презентаційних матеріалів, відеодоповіді та оформлення фінального звіту згідно з чинними стандартами ДСТУ. Під час виконання роботи необхідно продемонструвати вміння аргументувати доцільність обраних архітектурних рішень для вирішення конкретних технологічних задач.

3 ОПИС ВИКОНАНОЇ РОБОТИ
    3.1 Загальні відомості про систему
    Система Gemini є результатом передових досліджень Google DeepMind та Google Research, що представляє собою сімейство великих мультимодальних мовних моделей. На відміну від попередніх ітерацій ШІ, Gemini була спроєктована як natively multimodal архітектура. Це означає, що система тренувалася на різних модальностях, таких як текст, аудіо, зображення, відео та код, одночасно в межах єдиного навчального процесу. Такий підхід забезпечує значно глибше розуміння зв’язків між різними типами даних порівняно з системами, що використовують окремі енкодери. Практичне призначення Gemini охоплює широкий спектр завдань: від генерації складного програмного коду та семантичного пошуку до виконання автономних агентних дій. Архітектура підтримує три основні рівні продуктивності, включаючи конфігурації Ultra для критичних обчислень, Pro для загальних завдань та Flash для випадків, що потребують мінімальної затримки.
    3.2 Архітектурний стиль системи
    Архітектура Gemini побудована як багаторівнева розподілена система, що поєднує кілька фундаментальних підходів для забезпечення глобальної доступності. Сервісно-орієнтована архітектура (SOA) дозволяє системі оперувати як набір автономних сервісів із чіткими контрактами взаємодії, що підвищує загальну надійність платформи. Мікросервісний стиль забезпечує незалежне розгортання кожного функціонального компонента, що є критичним для підтримки CD/CI процесів у глобальних масштабах та ізоляції потенційних помилок. Cloud Native підхід гарантує повну інтеграцію з інфраструктурою Google Cloud, використовуючи спеціалізовані процесори TPU v4 та v5p для досягнення екстремальної паралелізації обчислень. Подієвоорієнтована модель через асинхронний обмін повідомленнями дозволяє ефективно обробляти ресурсоємні запити, такі як аналіз довгих відеофайлів, не блокуючи ресурси API-шлюзу.
    3.3 Основні компоненти системи
    Структура Gemini включає кілька критично важливих підсистем, що забезпечують повний цикл обробки даних. API Gateway виступає єдиною точкою входу, забезпечуючи автентифікацію розробників, контроль квот та інтелектуальну маршрутизацію трафіку. Request Orchestrator функціонує як центральний інтелект архітектури, що координує виконання запиту, взаємодіючи зі службами безпеки та системами векторного пошуку для отримання контексту. Model Serving Cluster представляє собою глобально розподілений кластер вузлів інференсу, де розміщені ваги моделей та виконуються основні обчислення. Safety Service є автономним контуром валідації, що аналізує кожен токен на відповідність етичним фільтрам. Сховища даних включають векторні бази для механізмів RAG та реляційні бази для збереження метаданих сесій.
    3.4 Взаємодія компонентів та життєвий цикл запиту
    Життєвий цикл запиту в архітектурі Gemini демонструє високий ступінь автоматизації та координації. Клієнтський застосунок ініціює запит, який надходить на шлюз для автентифікації та передачі оркестратору. Оркестратор ініціює паралельну перевірку безпеки та пошук додаткового контексту у векторних сховищах. Після формування розширеного промпту дані передаються до кластера моделей. Якщо модель потребує актуальних даних з мережі, вона генерує виклик зовнішніх інструментів через Tools API. Остаточна відповідь збирається оркестратором та транслюється клієнту у вигляді асинхронного потоку токенів. Такий підхід забезпечує мінімальний час першої реакції системи навіть при обробці складних багатоетапних завдань.
    3.5 Масштабованість та надійність
    Надійність Gemini базується на принципах еластичності сучасних хмарних систем. Масштабування реалізовано на рівні Kubernetes (GKE) через механізми Horizontal Pod Autoscaler, що дозволяє автоматично збільшувати кількість інференс-вузлів під час пікових навантажень. Балансування навантаження здійснюється через Google Cloud Load Balancer, який забезпечує географічний розподіл запитів до найближчого вільного дата-центру. Для кешування станів діалогів та збереження контексту користувача використовується високопродуктивне сховище Redis. Відмовостійкість системи гарантується реплікацією даних у кількох зонах доступності та автоматичним відновленням контейнерів у разі збою окремих вузлів обробки.
    3.6 Приклади програмної реалізації
    В ході аналізу було досліджено застосування архітектурного патерна Адаптер для інтеграції з Gemini SDK. Клас GeminiAdapter виступає проміжним шаром між офіційною бібліотекою та бізнес-логікою додатка. Це дозволяє команді розробників змінювати конфігурацію моделі або переходити на нові версії API без рефакторингу основного коду сервісів. Ізоляція клієнта від специфічних методів SDK значно спрощує написання модульних тестів та забезпечує дотримання принципів чистої архітектури. Практична реалізація демонструє, як за допомогою TypeScript можна створити стабільний інтерфейс для взаємодії з складними зовнішніми AI-платформами.


4 ВИСНОВКИ
     У ході виконання практичного заняття було проведено комплексне дослідження архітектури системи Gemini. Встановлено, що її ефективність базується на поєднанні мультимодального ядра та мікросервісної інфраструктури у хмарному середовищі. Аналіз показав критичну роль оркестрації запитів та вбудованих механізмів безпеки у забезпеченні стабільної роботи платформи. Практичне застосування патерна Адаптер підтвердило можливість створення гнучких та тестованих систем на базі LLM. Отримані знання дозволяють краще розуміти принципи побудови сучасних високонавантажених систем та застосовувати навички архітектурного аналізу при розробці власного програмного забезпечення.
